# 0.4 Beta Current Limitations

Status: active as of 2026-03-25

## Purpose

Honest accounting of what the 0.4 public-tryability beta does not yet do, what is intentionally narrow, and what requires operator workarounds. This is the beta-specific consolidation of docs/guides/known-limitations.md with 0.4 contract-specific gaps.

Read this with:
- docs/guides/known-limitations.md (general known limitations)
- docs/beta/public-tryability-milestone.md (current beta scope)
- docs/guides/verification-and-testing.md (verification path)

## Provider Breadth

The 0.4 beta intentionally ships with narrow provider coverage:

| What works end-to-end | What exists as manifest only |
|---|---|
| Gmail read-only watch | Slack, WhatsApp, GitHub, Drive, Calendar, Notion |
| SMTP reviewed send | |
| Local-directory retrieval | |
| Forward-email ingress | |
| n8n webhook handoff | |

This is deliberate. The product is strongest on the worker/review/governance loop, not on integration breadth.

## Product Surface Is Still Converging

Some lower-priority surfaces are still catching up to the worker-first product
shell:

- a few docs and labels still reflect older pilot-era or chat-first framing
- some setup and recovery paths are still script-first rather than fully guided
  in the UI
- some provider surfaces exist mainly as narrow reference implementations, not
  as broadly polished product areas

## Identity and Access

- No SSO / SAML / OIDC — password auth only
- No MFA / passkeys
- No self-service password reset — admin or database intervention required
- Invitation and people management UX is narrow compared to worker/setup UX

## Gmail and Email

- Gmail setup is not zero-config — requires Google OAuth app credentials
- Gmail is read-only; outbound email uses SMTP relay
- Gmail "configured" is not "actively monitoring" — requires worker attachment and explicit inbox check
- SMTP needs real server credentials for reviewed send to work
- Missing SMTP blocks reviewed send — approval stays pending until configuration is fixed

## Retrieval

- Retrieval exists (local-directory sync, full-text search, document chunking) but is smoke-level proven, not benchmark-proven
- Only used by runtime-worker in one spot
- Not exposed through control-plane routes
- No console UI for retrieval configuration or results
- No semantic search — full-text only

## Deployment and Operations

- Single-node only — no HA or clustering
- Self-hosted only — no managed/SaaS offering
- No built-in secret manager
- Basic readiness only (/healthz and /readyz, no full metrics/alerting)
- No automatic backups — operator must back up Postgres and object storage
- No published container registry images — built from source today

## Security Gaps

- No field-level encryption layer
- No cryptographically chained audit log
- No full sandbox enforcement for every tool execution path
- No mature production observability stack

## What This Beta Is

This beta proves:
- the governed worker loop works (ingest → context → decide → review → execute → activity)
- strangers can deploy and try the product without undocumented operator
  knowledge
- the product has honest first-value paths that don't require Google
- one real reviewed outbound side effect works end-to-end

It does not prove:
- broad provider/channel parity
- enterprise-grade security or compliance
- production-scale performance or reliability
- managed deployment readiness
